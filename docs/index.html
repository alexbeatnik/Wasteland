CORE PROTOCOLS

▣ Offline-First Inference happens entirely on your machine through llama.cpp. No cloud relay, no API key, no model phoning home. // llama.cpp · GGUF
⚠ Hard Network Lockdown Linux seccomp-bpf installs SCMP_ACT_KILL_PROCESS on socket(AF_INET, …), AF_INET6, AF_PACKET the moment a model is loaded. Cannot be undone for the lifetime of the process. // seccomp · Linux only
▶ Streaming UI · 60 FPS Nuklear immediate-mode GUI on SDL2 + OpenGL. Background inference thread emits tokens into a mutex-guarded ring buffer; UI drains it every frame. // Nuklear · SDL2 · pthreads
◈ Smart <think> Rendering Reasoning blocks (Qwen3-style) auto-detected, dimmed, labeled ▒ thinking, and excluded from copy-to-clipboard. Line-start guard prevents false positives from prose. // 7-byte carry buffer
∑ Summarising Compact Live CTX: used / max bar with colour-coded progress. [ COMPACT ] doesn't drop the oldest turn — it asks the model to write a 3–6 sentence note that preserves names, numbers, paths, decisions and open questions, then rewrites history as [SUMMARY]…[/SUMMARY] + the last 2 turns verbatim. Each subsequent compact re-summarises (existing summary + newly-aged turns), so the prefix stays bounded no matter how long the conversation runs. // async · re-summarisation · ~600 chars
⌬ Sandboxed Agent Mode Tool-using ReAct loop: read_file, list_dir, write_file, apply_edit. Four-button capability selector (OFF / READ / RW / CUST) gates which tools the model may invoke; the active button is amber-filled with black text so the choice is unambiguous. CUST exposes per-tool checkboxes for fine-grained policy. On Linux, a separate wasteland-agent-executor subprocess applies seccomp + Landlock before touching the filesystem. // 10-turn cap · approval gate · IPC executor
▤ Multi-Chat Persistence Named, browsable chat sessions. Auto-named from your first message, then refined by the model into a 3–5 word contextual title — the title prompt tells the model the budget (≤ 40 chars, language-matched, no markdown) so it doesn't run on. Stored with authenticated XChaCha20-Poly1305 encryption (WST2 header) via Monocypher — detects tampering, not just obfuscation. // 64 chats · cross-launch · authenticated
✓ Model Verification Background SHA-256 of any .gguf in the local vault. One click on the ✓ button runs an incremental hash in a detached thread; progress bar and green/red result confirm file integrity. // sha256 · incremental · background
☰ Platform Sandbox Status Persistent UI indicator shows available sandbox capabilities in real time: green when seccomp + Landlock are active, amber for partial, red for none. Never falsely claims security. // seccomp · Landlock · runtime-probed
↻ One-Click Auto-Update Background GitHub probe at startup (before lockdown). New release detected → orange banner. Click [ DOWNLOAD UPDATE ] → [ RESTART TO UPDATE ] → app exits, generated script replaces the binary (with pkexec / runas / osascript elevation when needed) and relaunches. // 5 s timeout · curl
⌨ Cyrillic + HiDPI DejaVu Sans Mono baked into the binary via xxd -i — covers Latin, Cyrillic (Ukrainian U+0400–U+04FF), and Geometric Shapes (U+25A0–U+25FF). Scales with display DPI on Retina, Windows 125%–200%, and Wayland fractional. // 343 KB · embedded

SECURITY MODEL

  1. App boots with the network reachable — you need it to download models.
  2. Nothing is loaded automatically. You pick a model from the local vault.
  3. On a successful [ LOAD ], lockdown_network() runs:
    • Linux: seccomp-bpf with SCMP_ACT_KILL_PROCESS on socket(AF_INET|AF_INET6|AF_PACKET, …). Filters argument family only — Unix-domain sockets (X11/Wayland) keep working. A Landlock ruleset further restricts filesystem access to the workspace directory.
    • macOS / Windows: no-op (platform limitation).
  4. seccomp filters cannot be removed for the lifetime of the process. Unloading a model does not lift the lockdown — restart to download more.
  5. Download / update code lives only in network.c. The UI hides the entire HUB section the moment state->network_lockdown is true.
  6. Agent tools on Linux run inside a separate wasteland-agent-executor subprocess that applies both sandboxes before entering its service loop, minimising filesystem surface even if the main process is compromised.
